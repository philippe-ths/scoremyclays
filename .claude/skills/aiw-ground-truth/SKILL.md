# Ground Truth

Read this file before generating, sourcing, or reusing any input or expected output that a test or validation will compare against. Read it again whenever the task's modality is unclear, when the agent is about to invent example data, or when reusing fixtures from earlier in the session.

## Why This Skill Exists

In AI-assisted coding the same actor writes the code, generates the test fixtures, runs the tests, and reads the results. Four roles, one actor, no external check. When everything passes, that fact alone proves nothing — the loop is self-referential. This skill keeps the loop broken by anchoring the agent's checks to inputs and expected outputs whose authority does not come from the agent.

A test cannot be more trustworthy than its oracle. If the oracle was invented by the same process that produced the code under test, a passing test is evidence only that the agent is internally consistent — not that the system is correct.

## Trust Hierarchy

Classify every input and expected output the work will rely on. Highest trust first:

1. **Real production data or captured runs from real usage.** Inputs from real users in real environments, captured during real execution.
2. **Real artifacts from the upstream tool or system that produces the inputs.** Files, payloads, or events generated by the actual upstream producer, not a model of it.
3. **Snapshots of previous known-good runs of the system itself.** A captured output from a prior version known to behave correctly, used to detect drift.
4. **Hand-constructed minimal examples confirmed by the user.** Small inputs and expected outputs the user has reviewed and explicitly stated represent real behaviour. "Looks fine" does not count; the user must affirm the example reflects what the real system does or should do.
5. **Synthetic fixtures generated by the agent.** Anything the agent invented. Useful for exercising code logic but not authoritative about what the system should do.

For every fixture or expected output the work relies on, record its trust level alongside it. Mark synthetic fixtures visibly wherever they appear — in filenames, headers, or inline comments — so no future read mistakes them for higher-trust data. (Why: trust level decays silently when the marker is missing; the next reader assumes provenance from absence of warning.)

When the trust level required for the task is not available, do not silently substitute a lower level. Apply the sourcing protocol below.

## Modality Decision Procedure

This section is the canonical modality reference for the workflow. Other skills cite it by name; do not redefine modality classification elsewhere.

### The nine modalities

- **New.** No prior code in this area. Building from scratch.
- **Feature.** Adding capability to existing code that did not have it.
- **Fix.** Correcting behaviour that is broken.
- **Refactor.** Changing structure while preserving observable behaviour exactly.
- **Improve.** Non-behavioural quality work: readability, simplification, naming.
- **Investigate.** Producing understanding rather than code. The output is what was learned.
- **Migrate.** Translating to a new substrate: library, framework, version, language.
- **Configure.** Wiring up external systems, integrations, or environment-specific setup.
- **Delete.** Removing code or capability.

### How to classify

For each task:

1. **Name the primary outcome the task asks for.** Pick the modality whose definition fits.
2. **Ask whether the task involves more than one of the nine outcomes.** If yes, the task is compound.
3. **For compound tasks, list every modality that applies.** Apply the union of all relevant rules. When two rules conflict, the stricter rule wins. (Why: a compound task inherits the failure modes of every modality involved; relaxing constraints for one risks regressions in another.)
4. **State the classification before any oracle decision.** The oracle depends on the modality.

If classification is genuinely ambiguous, name the candidate modalities and ask the user which fits. Do not pick the modality that minimises required ground truth.

### Common misclassifications to watch for

- **Fix misclassified as Feature.** Writing new code that paves over the broken behaviour rather than diagnosing what broke. The oracle for a fix is the bug itself; without it, the agent fabricates plausible new code that may not address the original failure.
- **Migrate misclassified as Refactor.** Preserving behaviour that existed as a workaround for the old substrate's limitations. Refactor's oracle says "previous behaviour is correct"; migrate may need to drop behaviour, not preserve it.
- **Delete misclassified as trivial.** "Just remove it" still has an oracle: the absence of remaining dependencies on what was removed. Skipping this leaves dead references that compile but fail at runtime.

## Modality-Specific Oracle Rules

The oracle is whatever defines "correct" for this task. The rule depends on the modality:

- **New.** Oracle is constructed examples confirmed by the user before code is written. Pull in adjacent real artifacts wherever any exist. The spec states requirements; it is not oracle.
- **Feature.** Hybrid. Constructed examples (user-confirmed) for the new capability. Characterisation of the surrounding code for what must remain unchanged.
- **Fix.** The bug report defines the oracle for the bug itself: what was observed, what should have happened. Pair with characterisation of everything else to detect collateral damage.
- **Refactor.** Captured behaviour of the current code, treated as authoritative for the duration of the task. Characterisation tests are the primary tool.
- **Improve.** Behavioural aspects: same as refactor. Non-behavioural aspects (naming, readability, simplification): no oracle exists. Say so explicitly rather than inventing one.
- **Investigate.** Oracle is reproducibility of the issue, not correctness of code. The output is understanding; the check is whether the agent can reliably reproduce or explain the observed behaviour.
- **Migrate.** Previous behaviour on real inputs, with an explicit exception list for behaviour that was a workaround for the old substrate's limitations and should not be preserved. The exception list must be enumerated, not implied.
- **Configure.** The real external system's actual behaviour. Mocks are not ground truth in this modality. (Why: the point of configuration work is to interact correctly with the real thing; a green test against a mock proves only that the agent's mental model of the external system is self-consistent.)
- **Delete.** Inverted. Oracle is the absence of dependencies on what is being removed. Checks should fail if something still calls or imports what was deleted, not pass.

For compound modalities, the oracle is the union of the relevant rules. If a refactor adds a feature, the agent owes both characterisation of the existing code and user-confirmed examples for the new capability.

## Sourcing Protocol

When the modality requires a trust level higher than what is currently available:

- Do not invent a substitute.
- Do not downgrade silently to a lower trust level.
- Stop and ask the user.

Use language that names what is missing, why it matters, and what would unblock the work. For example:

> "This task is a [modality]. The oracle should be [trust level / description]. I only have [what is available]. If I proceed with what I have, I cannot detect [specific class of regression]. Can you provide [specific artifact], or should I proceed with the lower-trust data and flag the gap?"

Proceeding with lower-trust data is acceptable only when the user has explicitly agreed and the gap is recorded with the work.

## Storage and Provenance

Ground-truth artifacts must carry their own provenance. Project conventions decide where artifacts live; the metadata travels with them regardless of location.

For every ground-truth artifact, record:

- **Trust level** (1–5 from the hierarchy).
- **Origin.** Where it came from: which user, which upstream tool, which prior version, which production capture.
- **Capture date.** When it was obtained.
- **Modifications.** Any change since capture, and why.
- **Staleness criteria.** What would make this artifact no longer trustworthy: upstream version change, schema migration, dependency upgrade, time elapsed.

Provenance lives either inside the artifact (header comment, sidecar file, frontmatter) or in a manifest co-located with it. Do not assume provenance can be reconstructed from filenames or directory structure alone. (Why: filenames drift, directories get reorganised, and the next reader has no way to recover the source.)

Regenerate ground-truth artifacts when their staleness criteria are met. Do not regenerate them to make a failing test pass — see anti-patterns below.

## Anti-Patterns

These patterns produce green tests with no real authority. Refuse them.

- **Treating the spec as ground truth.** A specification states intent. It is not evidence of the system's behaviour. Tests written against the spec alone confirm that the agent followed the spec, not that the system works.
- **Inline-generated fixtures presented as non-synthetic.** Building example data inside the test file and treating it as ground truth because it sits outside a `synthetic/` directory. Provenance is about source, not location.
- **Persisting synthetic fixtures across runs as if persistence made them trustworthy.** A synthetic fixture committed to disk is still synthetic. Time and version control do not promote it up the hierarchy.
- **Modifying ground-truth artifacts to make a failing test pass.** A failing test against real ground truth is a signal about the code, not the oracle. Changing the oracle silences the signal. If the oracle is genuinely wrong, mark it as wrong, replace it with a higher-trust source, and rerun the test.

## Boundary Between Ground Truth and Test Setup

Not every input a test uses is ground truth. The distinction is what the input represents:

- If the input represents something that exists in the real system — a real user, a real upstream output, a real configuration, a real event — the trust hierarchy applies. Source it from real production or upstream artifacts when available, hand-constructed examples confirmed by the user otherwise, and never the agent's invention alone.
- If the input is a synthetic shape used to exercise a function's branches, error paths, or boundary conditions — and does not represent something the real system would actually encounter — it is test setup. It lives inline with the test, is marked as such, and does not require provenance metadata.

When in doubt, ask: "if a real user ran the real system, could this exact input occur?" If yes, ground-truth rules apply. If no, it is test setup.

The aiw-testing skill consumes this boundary rule. Both skills must agree on which side of the line a given input lives.
